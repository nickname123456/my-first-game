# PM Simulator

**PM Simulator** - 2D top-down игра на `pygame`, в которой игрок управляет маленькой IT-командой перед релизом продукта. Игрок берет роль проджект-менеджера: назначает задачи, следит за дедлайнами, помогает сотрудникам, реагирует на кризисы и пытается довести проект до релиза без развала бюджета, качества и морали команды.


## Идея игры

Игрок находится в офисе стартапа. В офисе есть рабочая зона, кухня, переговорка, канбан-доска и сотрудники с разными специализациями. На доску приходят задачи с дедлайнами, сложностью, ценностью и требуемым навыком. Игрок должен назначать задачи подходящим сотрудникам и принимать решения, когда команда сталкивается с проблемами.

Игра длится ограниченное время. В конце происходит релиз, и результат оценивается по количеству выполненных задач, качеству продукта, оставшемуся бюджету, морали команды, доверию заказчика и уровню технического долга.

## Игровой процесс

В текущей версии игрок:

- запускает игровую сцену из главного меню;
- перемещается по офису с помощью `WASD` или стрелок;
- видит офисную карту, рабочую зону, кухню, переговорку и канбан-зону;
- открывает канбан-доску и назначает задачи сотрудникам;
- подходит к сотрудникам и решает их проблемы;
- выбирает, как реагировать на кризисы;
- балансирует между скоростью разработки, качеством, бюджетом и выгоранием команды;
- завершает сессию победой или поражением и видит экран результата со счетом и рекордом.

Сотрудники:

- имеют специализацию: backend, frontend, QA, DevOps, AI;
- автономно перемещаются по офису;
- выполняют задачи с разной скоростью и риском ошибок;
- устают, отдыхают, попадают в стресс и могут выгореть;
- могут просить помощи, если задача сложная или дедлайн близко.

## Основные ресурсы

- **Время до релиза** - главный ограничитель игровой сессии.
- **Бюджет** - тратится на помощь, переработки и исправление кризисов.
- **Мораль команды** - падает из-за перегруза, конфликтов и плохих решений.
- **Качество продукта** - зависит от правильного назначения задач и количества ошибок.
- **Технический долг** - растет при быстрых, но грязных решениях.
- **Доверие заказчика** - меняется из-за дедлайнов, качества и реакции на проблемы.

## Алгоритмы

В проекте используются алгоритмы, которые напрямую влияют на игровой процесс и могут быть показаны на защите не только в коде, но и в поведении игры.

### 1. A* для перемещения сотрудников по офису

**Зачем нужен:** сотрудники должны самостоятельно ходить к рабочему месту, канбан-доске, кухне, переговорке или к игроку. Простое движение по прямой не подходит, потому что в офисе есть стены, столы и непроходимые зоны.

**Как работает:** карта офиса представляется как сетка. Каждая клетка либо проходимая, либо заблокированная. Алгоритм A* ищет путь от текущей клетки сотрудника до целевой клетки, используя стоимость пути и эвристику Манхэттенского расстояния.

**Где используется:**

- сотрудник идет к рабочему месту после получения задачи;
- уставший сотрудник идет на кухню;
- сотрудник с проблемой идет к игроку или в переговорку;
- сотрудник обходит мебель и стены.

**Входные данные:** сетка офиса, стартовая клетка, целевая клетка, список препятствий.

**Выходные данные:** список клеток пути или сообщение, что путь недоступен.

**Сложность:** `O(E log V)`, где `V` - количество клеток, `E` - количество переходов между клетками. При использовании приоритетной очереди выбирается наиболее перспективная клетка.

**Псевдокод:**

```python
def astar(grid, start, goal):
    open_set = PriorityQueue()
    open_set.push(start, priority=0)
    came_from = {}
    cost_so_far = {start: 0}

    while not open_set.empty():
        current = open_set.pop()

        if current == goal:
            return restore_path(came_from, start, goal)

        for next_cell in grid.neighbors(current):
            new_cost = cost_so_far[current] + grid.cost(next_cell)

            if next_cell not in cost_so_far or new_cost < cost_so_far[next_cell]:
                cost_so_far[next_cell] = new_cost
                priority = new_cost + heuristic(next_cell, goal)
                open_set.push(next_cell, priority)
                came_from[next_cell] = current

    return []
```

### 2. Behavior Tree для поведения сотрудников

**Зачем нужен:** сотрудники не должны быть статичными объектами. Они должны выбирать действия в зависимости от ситуации: работать, отдыхать, просить помощи, идти на встречу, ускоряться перед дедлайном или уходить в выгорание.

**Как работает:** поведение сотрудника описывается деревом узлов. Есть условия, последовательности и выбор. Каждый кадр дерево выполняется и выбирает подходящее действие.

**Пример логики:**

- если сотрудник выгорел - ничего не делать;
- если есть кризис - искать помощь;
- если усталость высокая - идти отдыхать;
- если есть назначенная задача - идти работать;
- если задач нет - ждать или идти к канбан-доске.

**Где используется:**

- автономный выбор действия сотрудника;
- реакция на дедлайн;
- переход к отдыху при высокой усталости;
- просьба о помощи при проблеме.

**Входные данные:** состояние сотрудника, задача, усталость, дедлайн, наличие кризиса, положение на карте.

**Выходные данные:** выбранное действие сотрудника.

**Сложность:** `O(n)` за тик, где `n` - количество узлов дерева, проверенных до выбора действия.

**Псевдокод:**

```python
def employee_behavior(employee):
    selector = Selector([
        Sequence([IsBurnedOut(), DoNothing()]),
        Sequence([HasCriticalProblem(), AskManagerForHelp()]),
        Sequence([IsTooTired(), GoToKitchenAndRest()]),
        Sequence([HasTask(), GoToWorkplace(), WorkOnTask()]),
        Sequence([NoTaskAssigned(), GoToKanbanBoard()])
    ])

    return selector.tick(employee)
```

### 3. Priority Queue для планирования задач

**Зачем нужна:** на канбан-доске может быть много задач. Игроку нужна понятная рекомендация, какие задачи важнее закрыть первыми. Простая сортировка по одному параметру не подходит, потому что важны дедлайн, сложность, ценность, риск просрочки и соответствие навыку.

**Как работает:** для каждой задачи считается приоритет. Чем ближе дедлайн, выше ценность и выше риск просрочки, тем выше задача поднимается в очереди. Для хранения используется `heapq`.

**Где используется:**

- сортировка задач на канбан-доске;
- рекомендация следующей задачи;
- автоматический выбор задачи для сотрудника;
- расчет риска провала релиза.

**Входные данные:** список задач, текущее время, параметры задач, список сотрудников.

**Выходные данные:** задача с максимальным приоритетом или отсортированная очередь задач.

**Сложность:** вставка и извлечение задачи - `O(log n)`, получение самой важной задачи - `O(1)` для просмотра вершины кучи.

**Псевдокод:**

```python
def calculate_task_priority(task, current_time):
    time_left = max(1, task.deadline - current_time)
    urgency = 100 / time_left
    value = task.business_value * 2
    risk = task.difficulty * 3
    penalty = task.estimated_time

    return -(urgency + value + risk - penalty)


def build_task_queue(tasks, current_time):
    queue = []

    for task in tasks:
        priority = calculate_task_priority(task, current_time)
        heapq.heappush(queue, (priority, task.id, task))

    return queue
```

### 4. Цепь Маркова для настроения и выгорания

**Зачем нужна:** настроение сотрудников не должно меняться полностью случайно. Оно должно зависеть от нагрузки, дедлайнов, помощи игрока и кризисов. Цепь Маркова позволяет моделировать вероятностные переходы между состояниями.

**Состояния настроения:**

- `normal` - сотрудник работает стабильно;
- `tired` - сотрудник устал, скорость ниже;
- `stressed` - высокий риск ошибки и конфликта;
- `burnout` - сотрудник временно не может работать.

**Как работает:** у каждого состояния есть вероятности перехода в другие состояния. Эти вероятности изменяются игровыми факторами. Например, перегруз повышает шанс перехода `tired -> stressed`, а помощь игрока повышает шанс `stressed -> normal`.

**Где используется:**

- расчет состояния сотрудника каждый игровой интервал;
- влияние решений игрока на команду;
- появление выгорания не как случайного события, а как результата накопленного давления.

**Входные данные:** текущее настроение, нагрузка, дедлайн, помощь игрока, активные кризисы.

**Выходные данные:** новое настроение сотрудника.

**Сложность:** `O(k)`, где `k` - количество возможных переходов из текущего состояния.

**Псевдокод:**

```python
def update_mood(employee, context):
    probabilities = get_base_transitions(employee.mood)

    if employee.workload > 80:
        probabilities["stressed"] += 0.20

    if context.manager_helped:
        probabilities["normal"] += 0.25

    if context.deadline_is_close:
        probabilities["tired"] += 0.10
        probabilities["stressed"] += 0.10

    return weighted_choice(normalize(probabilities))
```

### 5. PRD для генерации кризисов

**Зачем нужен:** обычный `random` может давать неприятные серии: кризисы появляются несколько раз подряд или не появляются слишком долго. PRD, псевдослучайное распределение, делает события честнее для игрока: если кризиса давно не было, шанс постепенно растет; после кризиса шанс сбрасывается.

**Как работает:** у каждого сотрудника или типа кризиса хранится накопленный шанс. Базовый шанс зависит от сложности задачи, неправильного исполнителя, усталости, техдолга и близости дедлайна. Если кризис не произошел, накопленный шанс увеличивается. Если произошел - сбрасывается.

**Где используется:**

- генерация багов;
- конфликты в команде;
- выгорание;
- сбои инфраструктуры;
- проблемы из-за технического долга.

**Входные данные:** базовый шанс кризиса, счетчик неудачных попыток, задача, сотрудник, ресурсы проекта.

**Выходные данные:** кризисное событие или отсутствие события.

**Сложность:** `O(1)` для одного сотрудника за проверку, `O(n)` для всей команды.

**Псевдокод:**

```python
def calculate_base_crisis_chance(employee, task, stats):
    chance = 0.03
    chance += task.difficulty * 0.01
    chance += employee.workload * 0.001
    chance += stats.tech_debt * 0.001

    if employee.skill != task.required_skill:
        chance += 0.08

    if task.is_deadline_close:
        chance += 0.05

    return min(chance, 0.40)


def roll_crisis(employee, task, stats):
    base = calculate_base_crisis_chance(employee, task, stats)
    chance = min(base * employee.failed_crisis_rolls, 0.80)

    if random.random() < chance:
        employee.failed_crisis_rolls = 1
        return create_crisis(employee, task)

    employee.failed_crisis_rolls += 1
    return None
```

## Вспомогательные механики

Эти части нужны для игры, но не являются главными алгоритмами для защиты:

- проверка коллизий игрока с препятствиями через `pygame.Rect`;
- конечные состояния задачи: `todo`, `in_progress`, `done`, `failed`;
- подсчет финального счета;
- сохранение лучшего результата в `highscore.json`;
- отрисовка HUD, меню, канбана и диалогов.

## Победа и поражение

**Победа:** игрок доживает до релиза и сохраняет основные ресурсы выше критического уровня.

**Поражение:** игра завершается раньше, если бюджет, мораль, качество или доверие заказчика падают до нуля. Также поражение возможно, если технический долг достигает критического значения.

## Система счета

Итоговый счет считается после релиза:

```python
score = 0
score += completed_tasks * 100
score += tasks_done_in_time * 50
score += quality * 5
score += morale * 3
score += budget * 2
score += client_trust * 4
score -= tech_debt * 6
score = max(0, score)
```

Лучший счет сохраняется локально в `highscore.json` и показывается в главном меню и на экране результата.

## Тестирование

Алгоритмы должны быть покрыты отдельными тестами без запуска pygame-окна:

- A* находит путь вокруг препятствий;
- A* возвращает пустой путь, если цель недоступна;
- Behavior Tree выбирает отдых при высокой усталости;
- Behavior Tree выбирает работу при назначенной задаче;
- приоритетная очередь ставит срочную и ценную задачу выше;
- цепь Маркова корректно нормализует вероятности;
- помощь игрока повышает шанс улучшения настроения;
- PRD повышает шанс кризиса после неудачных бросков;
- PRD сбрасывает счетчик после возникновения кризиса;
- итоговый счет не становится отрицательным;
- рекорд сохраняется только при улучшении результата;
- условия победы и поражения срабатывают корректно.

## Критерии готовности

Проект считается готовым, если:

- игра запускается через `main.py`;
- есть главное меню;
- игрок может начать новую сессию;
- офис отображается на одном экране;
- игрок ходит по карте и не проходит сквозь препятствия;
- сотрудники перемещаются по офису через A*;
- сотрудники выбирают действия через Behavior Tree;
- задачи сортируются через приоритетную очередь;
- настроение сотрудников обновляется через цепь Маркова;
- кризисы генерируются через PRD;
- работает канбан-доска и назначение задач;
- работают диалоги с вариантами решений;
- HUD показывает время, бюджет, мораль, качество, техдолг и доверие;
- игра завершается победой или поражением;
- лучший счет сохраняется между запусками;
- основные алгоритмы покрыты тестами;
- игра держит минимум 30 FPS.
